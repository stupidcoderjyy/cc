# 3.7 分析表与冲突处理

#### 目标

从 LALR 合并后的状态和已计算的前瞻符生成 `ACTION`/`GOTO` 分析表，并通过 `LALRConflictHandler` 接口处理冲突。`BuildParsingTable` 自动调用，冲突处理器默认使用 `DefaultConflictHandler`（基于优先级/结合性），可通过 `SetConflictHandler` 替换。

#### 实现

1. 定义冲突处理基础设施（`lalr_conflict_handler.h`）
    1. `enum class ConflictAction { kShift, kReduce, kAbort }` — 移入-规约冲突的返回值
    2. `enum class ActionType { kError, kShift, kReduce, kAccept }` — 分析表动作类型
    3. `struct Action { ActionType type; int target; }` — `target` 在 SHIFT 时为目标状态，REDUCE 时为产生式 ID
    4. `LALRConflictHandler` 纯虚接口：`HandleShiftReduce`（接收 2 组优先级/结合性参数）、`HandleReduceReduce`
    5. `DefaultConflictHandler` 默认实现：高优先级胜出，同优先级 `kLeft` → `kReduce`、`kRight` → `kShift`、`kNonAssoc` → `kAbort`；`HandleReduceReduce` 默认返回 `-1` 抛异常

2. `BuildParsingTable()` 实现
    1. 从 `lalr_states_` 的 `transitions` 填充 `SHIFT`（目标为终结符/EOF）和 `GOTO`（目标为非终结符）
    2. 从 `lr0_lookaheads_` 按 `lr0_to_lalr_` 合并到对应 LALR 状态，归约项（`dot_pos == body.size()`）的每项前瞻符填充 `REDUCE`
    3. `[ROOT -> S·]` 且前瞻符含 `$` 填充 `ACCEPT`
    4. 冲突时回调 `conflict_handler_`（构造函数自动创建 `DefaultConflictHandler`）

3. `SetConflictHandler(unique_ptr<LALRConflictHandler>)` — 替换已有的冲突处理器

4. gtest 覆盖（`lalr_parser_parsing_table_test.cpp`）
    1. `NoConflictsWithDefaultHandler`：表达式文法 + 优先级设置后无冲突
    2. `CustomHandler`：替换自定义处理器验证
